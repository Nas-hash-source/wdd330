Chapter 10: Testing and Debugging

Errors, Exceptions, and Warnings

  • Exceptions: error that references what should be fixed.
  • Stack Traces: bunch of intertwined functions that leads to the origin that produces the error
  • Warnings: the program still runs but may cause problems later.

Importance of testing and debugging

Strict Mode: ('use strict';) allows to abide in the best practice for avoiding sloppy codes.

Often in a team, you cannot always assume that they will follow strict mode, so putting the 'use strict' in a self-invoked function is recommended.

Linting Tools:

JS Lint, JS Hint, and ES Lint improve a programming style guide, it can be useful in a team for a mutual convention.

Feature Detection:

Since Browsers are continually evolving, but some stays older. We need to check if a particular browser has a feature or not by using if block. Some library such as Modernizr facilitate this detection.

Debugging in the Browser:

alert() was widely used to track breakpoints, variables in the past, now the console offers diverse variety to debug your codes.

Debugging tools: the debugger keyword can be used to set breakpoints. Remember to remove them before shipping.

Error Objects:

  • const error = new Error() //Error object can be customized when there are Exceptions
  • EvalError: Error related to the global eval() function
  • RangeError: out of range error
  • ReferenceError: an item or a function does not exist
  • SyntaxError: error in the syntax
  • TypeError: error in the type of the value
  • URIError: problem related to encoding and decoding the URL
  • InternalError: error from the JavaScript engine such as too much recursion

Throwing Exceptions

The throw statements allows us to create our own exception. It's better practice however to throw an error object.

Exception Handling

If we suspect that a piece of code will result in an exception, try to handle it with try, catch and finally

Writing tests before any actual code.
